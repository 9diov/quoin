# Type Definition Documents self-identify via `_type: type` frontmatter

A Type Definition Document is a normal Document whose frontmatter contains the system Type Declaration `_type: type`. Integrations discover Type Definition Documents by scanning frontmatter for this sentinel — not by inspecting file paths or directory layouts. The Parser verifies the sentinel is present and rejects Type Definition Documents that omit it or declare a different value.

The Type Declaration value space is therefore two-valued:

- `_type: "[[TypeName]]"` — a regular Document declaring conformance to `TypeName`.
- `_type: type` — a Type Definition Document. The bare literal `type` is reserved as a *value* under the Type Declaration key.

The name `type` is **not** reserved as a Type Reference name. A user may author a Type Definition Document whose `TypeDefinitionDocumentIdentity.name === 'type'`; by convention, that Document is the **user-extensible meta-Type Definition Document**. The Parser does not treat it specially — it is parsed exactly like any other Type Definition Document. Integrations that want stricter constraints on Type Definition Documents (extra required Properties, extra required Sections) load this Document and validate other Type Definition Documents against it through ordinary Validation. When no such Document exists, the Parser's hardcoded baseline (`_type: type` + `## Schema`) is the entire contract.

Rationale: frontmatter is the only place the type system already owns. Path conventions (`types/`, `_types/`, `schema/`) vary across integrations — Obsidian vaults, Hugo content trees, and GitBook spaces all organize files differently, and forcing a path would tie discovery to one integration's habits. Frontmatter discovery also lets a single Document act as a Type Definition Document regardless of where it lives, which matters in vaults where users freely move files. Collapsing the meta-type into the same `_type: type` machinery (rather than reserving `type` as forbidden) avoids a parallel schema-of-schemas mechanism — extensibility falls out of the existing Document + Type Definition Document relationship used reflexively. Alternative considered: discover by path (rejected — couples Core semantics to integration-specific layout, breaks when users reorganize). Alternative considered: discover by a separate top-level key like `_kind: type-definition` (rejected — adds a new reserved key with no benefit over reusing the `_type` channel that already governs typing). Alternative considered: reserve `type` as a forbidden Type Reference name (rejected — loses the extensibility win and forces a separate mechanism for any future "all Type Definition Documents must X" requirement).
